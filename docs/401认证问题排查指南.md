# 401 Unauthorized 认证问题排查指南

## 问题描述
前端在请求删除蓝桥杯获奖者接口（DELETE /api/winners/:id）时，已正确携带了Authorization Bearer Token，但后端依然返回401 Unauthorized。

## 已修复的问题

### 1. 角色定义不一致问题 ✅
**问题**: 存在多个角色定义不一致
- 枚举定义: `Role.ADMIN = 'admin'` (小写)
- RoleGuard判断: `user.role === 'ADMIN'` (大写)
- 数据库种子: `role: 'ADMIN'` (大写)

**修复**: 统一角色定义为大写
```typescript
export enum Role {
  ADMIN = 'ADMIN',
  CZ_MEMBER = 'CZ_MEMBER', 
  COMMON = 'COMMON',
  EDITER = 'editer',
}
```

### 2. RoleGuard逻辑优化 ✅
**问题**: 角色检查逻辑过于简单，缺乏详细日志

**修复**: 添加详细日志和错误处理
```typescript
canActivate(context: ExecutionContext): boolean | Promise<boolean> | Observable<boolean> {
  const user = context.switchToHttp().getRequest().user as user;
  const requiredRoles = this.reflector.get<Role[]>('roles', context.getHandler());
  
  this.logger.debug(`User role: ${user?.role}, Required roles: ${requiredRoles?.join(', ')}`);
  
  if (!user) {
    this.logger.warn('No user found in request');
    return false;
  }
  
  if (!user.role) {
    this.logger.warn(`User ${user.userId} has no role assigned`);
    return false;
  }
  
  // 如果没有指定特定角色，默认检查是否为ADMIN
  if (!requiredRoles || requiredRoles.length === 0) {
    const isAdmin = user.role === Role.ADMIN;
    this.logger.debug(`Default admin check: ${isAdmin}`);
    return isAdmin;
  }
  
  // 检查用户是否具有所需角色
  const hasRequiredRole = requiredRoles.includes(user.role as Role);
  this.logger.debug(`Role check result: ${hasRequiredRole}`);
  
  return hasRequiredRole;
}
```

### 3. JWT策略增强 ✅
**问题**: JWT验证缺乏详细日志和错误处理

**修复**: 添加详细日志和错误处理
```typescript
async validate(payload: any) {
  this.logger.debug(`Validating JWT payload: ${JSON.stringify(payload)}`);
  
  if (!payload.sub) {
    this.logger.warn('JWT payload missing sub field');
    throw new UnauthorizedException('Invalid token payload');
  }
  
  const user = await this.prisma.user.findUnique({
    where: { userId: payload.sub },
  })
  
  if (!user) {
    this.logger.warn(`User not found for userId: ${payload.sub}`);
    throw new UnauthorizedException('User not found');
  }
  
  this.logger.debug(`User validated: ${user.username} (role: ${user.role})`);
  return user;
}
```

## 排查步骤

### 1. 检查环境变量
确保 `.env` 文件中正确设置了 `TOKEN_SECRET`：
```bash
TOKEN_SECRET="your-super-secret-jwt-token-key-here"
```

### 2. 运行调试脚本
```bash
cd guPengyu/app/server
node debug-auth.js
```

### 3. 测试认证接口
使用测试接口验证认证是否正常：
```bash
curl -X GET "http://localhost:3000/api/winners/test-auth" \
  -H "Authorization: Bearer YOUR_TOKEN_HERE"
```

### 4. 检查后端日志
启动后端服务并查看详细日志：
```bash
cd guPengyu/app/server
```

观察以下日志信息：

- Token验证过程日志
- 用户角色检查日志

### 5. 验证用户角色
检查数据库中用户的角色是否正确：
```sql
SELECT userId, username, role FROM user WHERE role = 'ADMIN';
```

### 6. 前端Token检查
确保前端发送的token格式正确：
```javascript

```

## 常见问题及解决方案

### 1. TOKEN_SECRET未设置
**症状**: 后端启动时显示 "JWT Secret configured: NO"
**解决**: 在 `.env` 文件中设置 `TOKEN_SECRET`

### 2. Token过期
**症状**: JWT验证失败，提示token过期
**解决**: 重新登录获取新token

### 3. 用户角色不正确
**症状**: 用户存在但角色不是ADMIN
**解决**: 更新数据库中的用户角色
```sql
UPDATE user SET role = 'ADMIN' WHERE userId = 1;
```

### 4. Token格式错误
**症状**: 前端发送的token格式不正确
**解决**: 确保token以 "Bearer " 开头

### 5. 数据库连接问题
**症状**: 用户验证失败，提示用户不存在
**解决**: 检查数据库连接和用户数据
